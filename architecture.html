Layer 2 CrewAI multi-agent orchestration

Three specialised CrewAI agents run in sequence — each with a single responsibility. No LangChain or LangGraph needed; CrewAI handles task chaining natively.

Query analyst
Parses intent · extracts brand, color, price, sort preference
Search strategist
Decides alpha blend · builds Pinecone metadata filter
Results ranker
Blends similarity + rating + reviews → relevancy score 0–1
Uses Gemini 2.5 Flash with a JSON output schema to convert raw text into a structured ParsedIntent object.

Input: "red Nike shoes under ₹5000 high rated"
Output: { category: "shoes", brand: "Nike", color: "red", price_max: 5000, sort_by: "rating", specificity: "medium" }
LLM JSON mode outputPydantic schema validation Brand + color extractionPrice range parsing Specificity: high / medium / low
Decides how to search based on query specificity, then constructs the Pinecone query parameters.

Alpha (semantic blend weight):
 · Exact model name (high specificity) → alpha = 0.2 keyword-heavy
 · Descriptive/vague (low specificity) → alpha = 0.85 semantic-heavy
 · Mixed (medium, default) → alpha = 0.6 balanced hybrid

Also sets top_k (20–50 candidates) and routes to the correct Pinecone namespace (electronics / shoes / stationery).
Alpha blend decisionNamespace routing Metadata filter constructiontop_k sizing
Takes top-K Pinecone candidates and re-scores each using a cross-encoder model for precision, then blends with social proof signals.

Scoring formula:
score = (1 - α) × similarity + α × (rating/5) + review_boost

where review_boost = min(review_count / 1000, 1.0) × 0.05

Runs two parallel retrieval strategies and merges their ranked lists using Reciprocal Rank Fusion (RRF).

Semantic path: query embedded → cosine similarity search in Pinecone
Keyword path: BM25 tf-idf scoring on product name + description fields

RRF formula: score = α × 1/(k + sem_rank) + (1-α) × 1/(k + kw_rank)
where k = 60 (damping constant) and α comes from the Search Strategist.
To enable true sparse BM25 in Pinecone, upgrade to a sparse-dense hybrid index. The starter template simulates keyword ranking via token overlap scoring.
Uses all-MiniLM-L6-v2 via sentence-transformers (384 dimensions, ~80MB, free local inference).

Called in two contexts:
 1. Index time — product composed text → vector → stored in Pinecone
 2. Query time — search query → vector → cosine similarity search

Product text composition format:
[brand] [name] [color] [subcategory] [key_features] [description]

The quality of this composition is the single biggest factor in semantic search accuracy.
all-MiniLM-L6-v2384 dims ~80MB model sizeLocal inference (no API cost)
Applies a cross-encoder model (Cohere Rerank or a local model) as a second-pass precision stage after Pinecone retrieval.

Why not use it for the full index? Cross-encoders see the query and document together (joint encoding), making them far more accurate than bi-encoder similarity — but also 10–100× slower. By applying it only to the top-K candidates already filtered by Pinecone, you get high precision without sacrificing throughput.

Recommended: Cohere Rerank v3 for hosted, or cross-encoder/ms-marco-MiniLM-L-6-v2 for local.
Improves NDCG@10 Reduces false positives in top results

Source of truth for all structured product data. Pinecone is a derived index — PostgreSQL is the canonical record.

Key tables: products, ratings, categories, click_events

click_events logs every result click with position, product_id, and query. This is used offline to compute NDCG@10 — comparing which positions users actually clicked against what the ranker returned.

Pattern: Reflection (from Agentic Design Patterns/Reflection.py)

After the Results Ranker scores candidates, an evaluator checks result quality. If average relevancy falls below a threshold (e.g. 0.4), the system adjusts parameters and retries:

 1. Results Ranker produces scored list
 2. Evaluator checks avg relevancy, diversity, category coverage
 3. If FAIL → adjust alpha ±0.15, relax filters, increase top_k by 20
 4. Re-run search (max 2 retries)
 5. If PASS → return results

New file: app/agents/evaluator.py — Pydantic schema EvaluationResult(status, avg_relevancy, feedback, adjusted_alpha)
Self-critiqueIterative refinement Max 2 retriesConfigurable threshold
Pattern: Routing (from Agentic Design Patterns/Routing.py)

Instead of one-size-fits-all alpha, route to category-specific search strategies:

 · Electronics → α = 0.3 (keyword-heavy, users search by exact specs)
 · Shoes → α = 0.8 (semantic-heavy, users describe aesthetics)
 · Stationery → α = 0.5 (balanced, price-sensitive)
 · Unknown → α = 0.6 (fallback, search all namespaces)

Specificity adjustments are applied on top of the category base alpha.
Per-category alphaSpec-matching for electronics Aesthetic search for shoesFallback routing
Pattern: Parallelization (from Agentic Design Patterns/Parallelization.py)

Semantic (Pinecone vector query) and keyword (BM25/token overlap) paths are independent — run concurrently:

semantic, keyword = await asyncio.gather(_query_pinecone(...), _keyword_search(...))

Impact: Eliminates sequential wait, reducing search latency by ~40%. Both results feed into the same RRF merge step.
asyncio.gather~40% latency reduction No code change to RRF merge
Pattern: Multi-Agent Handoff (from Agentic Design Patterns/MultiAgent.py)

Handles compound queries like "laptop and a bag for it":

 1. Query Analyst detects multiple intents (categories) in one query
 2. Splits into sub-queries, each with its own ParsedIntent
 3. Each sub-query handed off to a parallel search flow
 4. Results merged and presented as grouped sections

Uses the same handoff schema pattern as the Hotel/Restaurant agent example.
Multi-intent detectionSub-query splitting Parallel search flowsGrouped results
