Entry point for all user interactions. Sends structured search requests to the FastAPI gateway and renders a results grid with relevancy badges.

React search UI

Each search fires a POST /api/v1/search to the FastAPI gateway with the full SearchRequest payload. Results render as product cards showing: name, brand, price, rating, and a relevancy score (0–1) computed by the ranker agent.

Output: { category: "shoes", brand: "Nike", color: "red", price_max: 5000, sort_by: "rating", specificity: "medium" }

Decides how to search based on query specificity, then constructs the Pinecone query parameters.

Alpha (semantic blend weight):
 · Exact model name (high specificity) → alpha = 0.2 keyword-heavy
 · Descriptive/vague (low specificity) → alpha = 0.85 semantic-heavy
 · Mixed (medium, default) → alpha = 0.6 balanced hybrid

Also sets top_k (20–50 candidates) and routes to the correct Pinecone namespace (electronics / shoes / stationery).
Alpha blend decisionNamespace routing Metadata filter constructiontop_k sizing
Takes top-K Pinecone candidates and re-scores each using a cross-encoder model for precision, then blends with social proof signals.

Scoring formula:
score = (1 - α) × similarity + α × (rating/5) + review_boost

where review_boost = min(review_count / 1000, 1.0) × 0.05

Runs two parallel retrieval strategies and merges their ranked lists using Reciprocal Rank Fusion (RRF).

Semantic path: query embedded → cosine similarity search in Pinecone
Keyword path: BM25 tf-idf scoring on product name + description fields

RRF formula: score = α × 1/(k + sem_rank) + (1-α) × 1/(k + kw_rank)
where k = 60 (damping constant) and α comes from the Search Strategist.
To enable true sparse BM25 in Pinecone, upgrade to a sparse-dense hybrid index. The starter template simulates keyword ranking via token overlap scoring.
Wraps OpenAI's text-embedding-3-small model (1536 dimensions, best cost/quality ratio for product text).

Called in two contexts:
 1. Index time — product composed text → vector → stored in Pinecone
 2. Query time — search query → vector → cosine similarity search

Product text composition format:
[brand] [name] [color] [subcategory] [key_features] [description]

The quality of this composition is the single biggest factor in semantic search accuracy.
text-embedding-3-small1536 dims Batched upsert (100 per call)Retry with exponential backoff
Applies a cross-encoder model (Cohere Rerank or a local model) as a second-pass precision stage after Pinecone retrieval.

Why not use it for the full index? Cross-encoders see the query and document together (joint encoding), making them far more accurate than bi-encoder similarity — but also 10–100× slower. By applying it only to the top-K candidates already filtered by Pinecone, you get high precision without sacrificing throughput.

Recommended: Cohere Rerank v3 for hosted, or cross-encoder/ms-marco-MiniLM-L-6-v2 for local.
Improves NDCG@10 Reduces false positives in top results

Layer 4 Storage

Three stores with distinct roles — vectors for similarity search, structured product data, and a query result cache.

Pinecone
1536-dim product vectors · 3 namespaces · metadata filters
PostgreSQL
Product catalog · ratings · click events · source of truth
Redis
Query result cache · TTL 5 min · session data
Namespaced by product category so searches stay scoped: electronics, shoes, stationery.

Each vector carries rich metadata for filter-at-query-time:
product_id · name · brand · category · subcategory · color · price · rating · review_count · stock_status · image_url

Index spec: dimension = 1536, metric = cosine, cloud = AWS, serverless.
Namespace per categoryMetadata filter at query time Serverless indexUpsert on product change
Source of truth for all structured product data. Pinecone is a derived index — PostgreSQL is the canonical record.

Key tables: products, ratings, categories, click_events

click_events logs every result click with position, product_id, and query. This is used offline to compute NDCG@10 — comparing which positions users actually clicked against what the ranker returned.
Caches full SearchResponse objects for repeated or near-identical queries, bypassing the full agent pipeline.

TTL rules:
 · Default queries → 5 minutes
 · Sort by rating → 30 seconds (ratings change frequently)
 · Cache key = hash(query + filters + sort_by)

Typical cached response: <10ms vs 300–800ms for a full pipeline run.

Traces every agent pipeline run end-to-end. Each span captures:
 · Query analyst LLM call (input prompt, output JSON, latency)
 · Search strategist decision (alpha, filters, namespace)
 · Pinecone query (vector, filter, top_k, result count)
 · Reranker call (candidates in, ranked list out)
 · Total pipeline latency

Essential for debugging degraded relevancy — you can replay any request and inspect every agent's exact inputs and outputs. Connect via LANGCHAIN_TRACING_V2=true in .env.
FastAPI exposes GET /api/v1/metrics (Prometheus format), scraped every 15 seconds.

5 tracked KPIs (per project requirements):
Search relevancy score — avg cross-encoder score per hour Click-through rate — clicks / impressions per query session Query processing time — p50 and p95 in ms Index update latency — PG write → Pinecone confirm delta Cache hit rate — Redis hits / total requests

Add a second Pinecone namespace per category storing CLIP image embeddings (openai/clip-vit-large-patch14, 768 dims). A user can upload a photo of a shoe and retrieve visually similar products — no text query needed. Product images stored in S3, served via CloudFront.
Pipe voice input through OpenAI Whisper (STT) to produce a text transcript, then feed that directly into the existing Phase 1 search pipeline — no changes needed to the agent layer or Pinecone index. Minimal integration effort for a high-value UX upgrade.
Swap the embedding model for a multilingual variant (e.g. multilingual-e5-large) and fine-tune on category-specific terminology. Evaluate translation quality using BLEU score comparing LLM-generated descriptions against Google Translate baseline — per the Capstone 4 methodology.
